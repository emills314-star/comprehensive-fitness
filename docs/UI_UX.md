## Experience principles

### Guided mesocycle workflow — IMPLEMENTED

The Templates hierarchy begins with **Plan Your Mesocycle**, followed by ordinary templates and Historical Mesocycles. Opening it creates a focused workspace: Before You Build guidance; objective/schedule/equipment/scope setup; empty training days; prioritized muscle needs; target-specific exercise search; focused pending set configuration; live workload review; Check Viability; exception acknowledgment; and final linked-template review.

One day is expanded at a time on mobile. Each assignment provides a working-set stepper, explicit Move to Day selector, and Remove action. Drag-and-drop is not required. The weekly summary distinguishes direct, fractional, and exposure-frequency values. Programming Guide remains available without discarding progress. The same exercise can be selected on multiple days without warning.

Warnings identify the affected day or muscle, explain why it matters, and suggest a correction. Passed checks and low-value notes remain hidden. Non-blocking exceptions may be accepted and remain audited in the draft.

Guide, Setup, Build, Check, and Create are compact step buttons. Active uses strong blue; completed/unlocked uses a light-blue return state with a checkmark; future steps are muted and disabled. Progress persists in the draft. Setup or Build edits retain compatible work but stale the prior viability result and relock Create.

Build keeps a compact Volume Remaining panel available. It shows direct/fractional sets, evidence-adjusted target range, sets to minimum, frequency need, remaining-day feasibility, and Below/Within/Above status. Candidate selection creates a pending Configuring Now card, focuses it with reduced-motion support, and shows target muscle, target-muscle effectiveness, confidence, sets, reps, RPE/RIR, structure, rest, and live warnings. Add to Day returns focus to the picker, updates needs immediately, and disables the same canonical exercise for that day while leaving it available on other days.

The current UI is mobile-first, light/dark capable, dense enough for training use, and designed to keep recommendations explainable. The implemented **Quiet Coach workspace** is a top-down structural redesign, not a palette layer over the previous five-tab UI. Rich navy owns primary actions, active navigation, focus, and selected state; success, warning, rest, and destructive colors remain semantically separate.

The modular layout is structural rather than user-configurable: users cannot drag, resize, or hide modules. Today leads with one coaching decision and quick starts. An active workout is one continuous document containing every exercise and set in program order; completing a set or rest interval updates the current-set highlight without moving the viewport, so the user remains in control of scrolling. An explicit exercise jump may move to the selected card. Readiness, notification state, history, and workout details are supporting disclosures. Plan owns templates and the mesocycle builder. Progress owns Overview, Lifts, and History in one workspace. More owns setup, privacy, data, and application controls.

Build separates unresolved **Needs Attention** items from a collapsed **Completed** summary. Status copy shows total effective sets first, then direct/fractional detail and frequency. Amber “Needs Frequency” cannot use the success treatment even when volume is within range. Create review uses expandable day cards with order, sets, rep/RPE/rest targets, volume contributions, warnings, and an Edit Day route. Successful creation remains on a persistent **Mesocycle Completed** panel with exact created/updated template counts and next actions; it does not return to Build.

Interaction principles evidenced in code: one canonical active workout; explicit destructive/final actions; immediate visible set updates; preserved drafts; conservative empty states; full recommendation detail on demand; accessible labels/live regions/inert modal backgrounds; and user-controlled overrides with reasons.

Pain-blocked exercise options list only current catalog-backed substitutes. Applying the explicit pain-free confirmation resolves the selected display name against that exact catalog set; an alias collision or duplicate normalized name is rejected instead of silently choosing another exercise.

## Design system

`index.html` is the authoritative implemented design-system source. Theme-level color variables in `:root` and `[data-theme="light"]` define background, panel, line, text, action, current, success, rest, warning, and destructive roles. Ten `[data-color-package]` layers remap those semantic roles while light/dark remains an independent display choice. Truthful `--color-*` aliases map established roles without changing their meaning. `coach-lead`, `active-workout-hero`, `insight-module`, and `progress-workspace-header` provide the shared hierarchy; `ChoiceChip`, `ChoiceTile`, and `DialogSheet` remain compatibility classes over established controls. Superseded `app-module`, `quiet-coach-hero`, and five-tab renderers are removed and guarded by stale-contract tests; session-board/support-drawer presentation is retired from the active Today renderer.

Ten livelier color packages are **IMPLEMENTED** and documented in `docs/design/UI_COLOR_PACKAGES.md`. Signal Garden is the default. Users select and persist a package in **More → Appearance and units → Color package**; the selected package changes cards, action/current, coaching/rest, success, warning, and destructive roles. Every non-Night-Stadium light package now uses a true-white application canvas instead of a tinted paper gradient; Night Stadium remains the intentionally dark package. Two five-screen Strong reference restyle sets remain local under the privacy-ignored `docs/design/strong-ui-mockups/` path because their source screenshots contain personal workout details.

The interface uses one compact system font stack, sentence-case labels, uppercase eyebrow labels, 6–18 px corner radii according to component scale, and dense 4–18 px spacing. Cards use a surface, one semantic border, and at most one status accent. Equivalent actions must reuse the existing primary, secondary, mini, text, destructive, selected, disabled, hover, pressed, and focus-visible patterns. Fixed navigation and transient notices must account for bottom safe-area insets and must not cover the final scrollable content.

Controls should provide a 44 px touch target where layout permits. Equipment `ChoiceChip` controls and Muscle Scope `ChoiceTile` labels meet that contract and expose programmatically named groups. Existing compact controls below 44 px are **NEEDS REVIEW** and must not be copied into new patterns. Text must wrap by default; a redundant narrow-header context may be visibly abbreviated only while its complete value remains in an accessible label and the full screen heading. Motion and programmatic scrolling must honor `prefers-reduced-motion`.

## Navigation and screen inventory

The persistent bottom navigation has four destinations (`primaryTabIds`, `render`):

| Tab | Purpose and major screens |
| --- | --- |
| Today | Quiet Coach next action, quick starts, a single-column active exercise/set logger, timer, submit confirmation, and completion summary. Readiness is captured before workout start rather than repeated in a bottom-of-workout adjustment panel. |
| Plan | Full template list, template cards, mesocycle planner/candidate pools, and readiness start sheet. |
| Progress | Overview for weekly volume/fatigue, Lifts for charts/recommendations, and History for submitted sessions and editing. |
| More | Units, goals/profile, readiness baseline, timers/notifications, PWA setup, workout-upload consent, bounded imports/export, remote deletion, and clear-local-data flow. |

The header exposes the current context, an atomic lb/kg switch, and a light/dark switch. More → Appearance and units exposes the ten-package color selector. Full template access is available from Plan; quick-start cards are a subset. Unit changes convert app-owned load values together while private source evidence remains unchanged and is converted only at the display/prescription boundary.

The first keyboard stop is a hidden-until-focused **Skip to main content** link. `#main-content` is a programmatically focusable route-entry target. Initial load leaves focus alone; an explicit selection of any of the four destinations moves focus into the newly rendered main view. Focus restoration across HTML-string rerenders uses an allowlisted, data-only `data-action` descriptor rather than storing detached nodes or accepting arbitrary selectors. Quick-template cards remain native buttons in the computed accessibility tree.

## Primary workflows

### Template and active workout

1. Select a quick start on Today or open the complete Plan destination.
2. Open the start sheet and enter/confirm readiness data.
3. Review adjustments and explicitly start.
4. Log set weight/reps/RPE; mark warm-up/type; complete or skip; add/reorder exercises/sets; view prescription/evidence.
5. Completing an eligible set can auto-start that exercise’s rest target. The resting set keeps its normal neutral boundary and shows a compact blue progress rail with the remaining time. Tapping the rail expands Pause/Resume, Cancel, −15, +15, and Skip controls; those controls do not consume workout space while the rail is collapsed. The notice names the next set and supports dismissal/return.
6. Request submit, review confirmation, then confirm final submission.
7. View post-workout grade, completed exercise results, PRs, highlights, and improvements.

Only one active workout is allowed. Other starts are disabled with one canonical resume notice (`scripts/test-performance.js`). Confirmed submission is guarded before any calculation with one shared acceptance predicate and a per-session in-progress lock. The click router consults that predicate before generic success feedback but still invokes the idempotent submission entry point, so double-routed or reentrant confirmation produces one completed workout, one accepted feedback sequence, and one set of effects. Template edits during a workout remain session-specific unless the user explicitly updates the template.

The active Today page is one compact workout column with no parallel Workout Board or bottom recovery-adjustment section. Each exercise places the deload toggle beside a collapsed resistance-type selector whose summary uses two rows: **Resistance** above the selected type, with a blue disclosure arrow. The recommendation summary names the actual generated role rows (for example, top and back-off loads with their exact role ranges), and expanded **Why This Recommendation** repeats those same rows with prior performance, today's target, confidence, and progression gate. Both are rendered from the set objects that prepopulate the editable Load, Reps, and RPE controls—not from a parallel generic summary. The expanded rationale also states the latest-history/high-watermark load anchor and explains when an old top performance is being approached conservatively. `basePrescription` remains explanation-only.

When **Previous** opens a submitted workout while another workout remains active, a safe-area-aware floating text-only **Return to active session** button stays above bottom navigation. It restores the current workout without editing either session. **Exercise options** separates **Research standard**, **Today’s guideline**, and **Saved default**, adds a Within/Below/Above advisory badge with dimensional differences, and uses a compact role-aware workload editor. Straight sets use one shared card; top-set/back-off prescriptions use visually distinct Top set and Back-off sets cards with independent counts and rep ranges, while multiple top sets use one repeated-primary-effort card. Applying a change uses the audited override path and rebuilds only unfinished working rows without collapsing their roles; completed sets, warm-ups, and actual repetitions remain untouched. An opt-in template default preserves the same shared or role-specific shape and is unchecked by default. Rep inputs remain freely editable above the displayed ceiling, and the copy explains that actual extra reps remain future progression evidence.

Adding a catalog-recognized exercise immediately renders its recommendation and prepopulated rows. Adding an uncatalogued exercise instead renders **Recommendation setup incomplete** inside Exercise options with the exact missing required metrics: primary muscle group, resistance mode, exercise style, progression metric, and applicable smallest increment. The exercise remains loggable. Completing those fields generates bounded custom guidance for unfinished work and displays the limitation on canonical ranking, biomechanics, substitution, and equivalence. Secondary muscle is optional. Each exercise also exposes **Did execution stay controlled?** with Controlled, Breakdown, and Not assessed; the submit confirmation repeats these controls, permits unassessed submission, and explains that missing assessment cannot confirm progression. The immediate completion summary adds a non-persisted **Base next exposure** preview and readiness caveat.

Working sets use one **Previous** column beside Load, Reps, and RPE; that cell combines the comparable load × reps @ RPE and its date instead of repeating history below three inputs. When comparable history exists, **Previous** is a native link to that exact submitted session in Progress History; returning to Today does not replace or submit the active workout. The Previous cell uses dedicated label, value, and date rows: the label stays in the header position, the compact dark-gray prior load/reps/RPE line shares the active fields' vertical centerline, and the date stays in its own lower row without crowding either line. The narrow set grid always reserves a status column, so long prior-performance text wraps inside the set boundary. Secondary per-set controls sit behind a 44 px ellipsis disclosure, so a collapsed future set stays near the reference row height while Rest, Skip, Remove, and **Progress when** remain available on demand. The add-set, add-warm-up, copy-set, and save-template actions use low-chrome controls with 44 px targets. Exercise options, add exercise, workout details, and progression rationale remain available through native disclosures so supporting controls do not dominate page height.

Template sets, reps, target RPE, load increment, and rest fields expose native `required` plus exact HTML min/max/step constraints backed by the same domain model used by patches and imports. Empty input therefore has browser `valueMissing` semantics in addition to the connected custom draft error. Invalid raw input stays visible in its connected field, receives `aria-invalid`/`aria-describedby`, and creates one card-level alert; it is not committed, saved, or erased by an unrelated rerender. Correcting the field clears the alert and creates one logical revision even when the browser delivers `input`, `change`, and `focusout`. Deleting the exercise/template, importing, or clearing local data removes any associated transient draft.

### Settings privacy and data controls

- Lock-screen notifications and cloud workout upload are separate controls. Upload defaults off and is enabled only by its own checkbox. During authorization the control is disabled, marked busy, and restored to its canonical value; it renders enabled only after server consent and the explicit IndexedDB app-data write succeed. Turning it off immediately invalidates old upload work, aborts in-flight requests, clears the pending queue, and deletes retained server workout copies. Offline revocation stays visibly pending for retry.
- Backup and personal-evidence import announce an attempt-specific pending, success, or error state in a polite live region. Full app backups accept up to 50 MiB so an exported large history/raw Strong archive remains recoverable; personal-evidence packages retain their separate 8 MiB limit. Invalid, oversized, overly deep/wide, executable-key, duplicate, orphaned, out-of-domain template/set/readiness-baseline, unsafe-revision, or invalid active/template recommendation-snapshot data is rejected before persistence without replacing current data. Snapshot rejection includes stale checksums, unsupported schemas, unresolved/colliding identities, unrelated exact targets, and host-exercise mismatches. Accepted backup revision metadata is rebased to the next local revision rather than trusted.
- When two valid saved copies are conflict-preserved, replacement import is disabled until confirmed Clear All. Export states that it contains the selected copy only and excludes the preserved alternate, so the user is not misled into treating one download as both copies.
- **Delete Remote Installation Data** is a separate Danger Zone action that preserves local workouts and resumes bounded server cleanup. Confirmed **Clear All Local App Data** also revokes workout consent and deletes the remote installation before discarding its bearer; it pauses offline or while remote cleanup or active-timer cancellation is unresolved. After confirmation it clears both app-data stores, writes fresh defaults, removes the conflict alternate again, and announces success only after the durable write.
- Exported app JSON remains the only implemented user-managed recovery source. Cloud workout upload is write-only and must never be labeled restore, cloud history, or an account. IndexedDB is primary, but failed-write recovery can temporarily place the complete app-data copy in localStorage, including workout history, settings, and imported personal-evidence aggregates; treat both stores and exports as sensitive same-device data.

### History and progress

Submitted sessions alone appear in history and analytics. History cards show a full wrapping blue session title, separate date, and a standalone color-coded grade letter with accessible combined labeling. Visible grade copy is `B`, `A-`, and so on—never `Grade B`. A submitted workout opens read-only in Today; editing requires an explicit mode and Save Edits confirmation, then PR/grade recalculation. History changes remain an in-memory transaction until that explicit Save. Numeric field input mutates the temporary model without replacing the DOM, so blur does not swallow the user's first Save Edits click; one real change advances one revision, and duplicate browser events do nothing. Primary-tab navigation and browser Back both open the same cancel-edit confirmation while keeping the edited submitted-workout view and canonical `#today` URL in place. Reload, page hide, and unload do not persist the temporary transaction.

Progress Overview drill-down is a focus-aware stack. Opening a root or nested volume, fatigue, history, or session detail focuses its Back control; Back restores the prior detail level, scroll position, and durable control that opened it. Leaving Progress clears the stack.

Charts are exercise-specific and interactive. Partial search is lexical over the cached catalog/history inventory, remains case-insensitive, shows all matches, and promotes the last explicit choice; it does not run progress analysis for each candidate or each keystroke. Selecting a point opens session-level details and surrounding context (`renderChart`, `renderChartPointDetail`) using an opaque key scoped to that render's exact exercise/window/date/resistance payload rather than recomputing an unscoped point from DOM values. Long names wrap in history/templates and suggestion controls; **NEEDS REVIEW:** physical testing should cover every chart axis/selector and very narrow devices.

At 200% text on a 320 CSS-pixel viewport, active set rows reflow from the compact six-column logger into two columns, with Previous spanning the row. Numeric controls suppress decorative spinner chrome, retain their full value area, and progression copy wraps instead of clipping.

The Charts scope controls form one rounded panel. It shows the canonical selected exercise, analysis type, qualifying-window label, exact dates, qualifying weeks, skipped deload weeks, and recalculation state. Exercise search and the custom period menu replace browser-default period dropdowns. Changing either clears dependent charts, score, rationale, expectations, actual-versus-expected, recommendations, and point detail until recalculation completes.

Charts now live under **Progress → Lifts**. A shared destination-level render boundary covers both Today and Progress. It clears only derived entity/analysis caches and retries once before recording a bounded destination/error code and rendering the error surface. Retry repeats that non-mutating rebuild. Recommendation presentation is independently bounded before that destination fallback: current-equipment or other hard rejections render as an in-context unavailable card, while older partial saved snapshots receive a display-only compatibility projection. The projection supplies safe labels and absent optional metrics without rewriting the stored audit snapshot. Legacy prescriptions may also store their explanation as a structured object; Today extracts bounded display text before rationale string operations and retains the resistance-aware prior-set formatter used by the expanded role rows. A malformed historical exercise label is excluded from the chart catalog instead of failing the whole Progress view. When an installed update is waiting, the error surface and banner expose **Update app/Update now** even during an open workout; the existing persistence gate saves the stable workout snapshot before reload and refuses activation if neither local store succeeds. Today additionally returns to its normal home screen when a stale route/runtime pointer has no valid session, before any session fields are read. Legacy `#charts`, `#dashboard`, and `#lift` links are compatibility redirects into Progress or Today; they are not primary destinations.

### Mesocycle planner

Mesocycles intentionally remain inside Plan with templates instead of becoming another destination. The guided workflow uses five dependency-ordered steps: Guide, Setup, Build, Check, and Create. The former eight-stage automatic planner and Program Slot UI are legacy historical-plan behavior, not the new-plan workflow.

Templates are progressively disclosed for responsiveness. The initial tab shows compact template rows, a compact current-mesocycle summary, and compact history. “Edit template” constructs that template's exercise controls only while open; “Open Planner Review” constructs the candidate, portfolio, session, and validation detail only while requested. Weekly fatigue warnings remain available on Dashboard, and readiness-adjusted coaching is calculated when starting a workout rather than repeated across every template row. Muscle-scope checkboxes update their local draft immediately without rebuilding the full tab after each tap.

The Muscle Groups in Scope control lists every consolidated available group and defaults to all selected. Unselected controls use a light/transparent neutral surface rather than a heavy filled state. The user may exclude any major or smaller group. The draft confirmation identifies every omission, offers Add to Mesocycle, and requires Keep These Exclusions and Continue before saving the draft as planned. `Why Train This Muscle Group?` uses the research database's anatomical functions plus muscle-family-specific training consequences; it does not repeat a generic omission explanation. Major groups retain a stronger badge, but the UI does not coerce the user after confirmation.

Candidate hierarchy is recommendation badge, exercise name, user-facing role, Predicted Program Effectiveness, Confidence, target muscles, then programming detail. `Why This Recommendation` is the shared expandable detail pattern for role, selection reason, progression, deload/rotation, replacement, and score factors. Internal enum/ID strings must be formatted through `presentationLabel`; raw `mg_*`, snake_case roles, confidence keys, and session IDs are not user copy.

Workout recommendation detail also uses the exact `Why This Recommendation` heading. It shows prior versus today, each material set/load/rep/RPE change, the evidence window, expected duration, and resumption criteria. Recommendation badges describe the actual delta (`Progress Load`, `Add One Rep`, `Hold`, or fatigue reduction), rather than using generic Progress when the prescription is unchanged. Pound fields and displays never expose floating-point conversion residue and use 0.5-lb increments.

Every active working-set card shows **Previous** from the newest eligible submitted workout with its date. Prior load, reps, and RPE are combined into one compact cell beside the current editable fields. This lookup searches the full submitted archive, not only the six-month analytics window. Matching uses stable performance identity first and exact Strong exercise name when identity provenance changed, while retaining resistance type and keeping differently named variations separate. It then prefers the same set role and role index, followed by the same ordered working-set position so imported Strong classification differences do not turn real history into an empty state. A compact dash is used only when no comparable set exists.

The review shows named session purpose, ordered exercises, roles, sets/reps, direct/indirect volume, target versus actual frequency, high-fatigue compounds, spinal/grip/systemic/local fatigue, duration, placement, substitutions, and severity-labelled findings. Blocking volume, frequency, or consecutive-heavy-pattern findings prevent activation. Historical plans show dates and lifecycle at the bottom of the tab. Never-activated drafts/plans may be deleted after confirmation; completed plans are protected and can be archived. Template cards show `Base Session Intent` only when a stable heavy/light/technique/deload/specialization intent exists and state that today’s readiness may modify execution.

An imported Strong exercise with exact submitted history remains usable when research identity, muscle target, or dynamic-target generation is unavailable. Its card says that the latest imported Strong performance and saved workout structure are being used and that research-based progression is unavailable until mapped; it does not present those non-safety taxonomy gaps as safety warnings. All 23 explicitly research-mapped Strong labels render the ordinary research-backed recommendation instead of an `unknown_exercise_identity` unavailable card, while the remaining custom names keep the explicit history fallback. Alias-backed recommendations retain each exact performance-history identity even when multiple names share one research exercise. **+ Set**, **+ Warm-up**, **Duplicate**, and each row's **Remove** control remain available during the active workout unless a real illness/pain/safety restriction applies. Set-count edits persist with the active workout and change the template only after **Update template from today**.

### Recovery and nutrition

### Legacy automatic mesocycle history

Legacy automatically generated plans remain readable through the compact Historical Mesocycles list and retain lifecycle actions that are valid for their saved status. The unreachable former eight-step automatic-planner review, candidate comparison, regeneration, and scope-confirmation UI has been removed. New guided plans do not automatically generate a portfolio or distribute sessions.

Available Equipment has an explicit `All Equipment / Standard Gym` default. The only other user-facing capability choices are Bodyweight, Bands, Dumbbells, Barbell, Rack, and Cable Station. Selecting an individual capability clears Standard Gym; clearing the final individual capability restores Standard Gym, so an empty ambiguous state is impossible. Restrictions apply to selected candidates, alternates, and comparisons; detailed requirements remain in the engine and are not exposed as a long picker. Omitted muscle groups use collapsed `Why Train This Muscle Group?` education. Portfolio entries prioritize exercise name, target muscle, program role, and primary/secondary designation without repeating generic boilerplate. Primary, secondary, and destructive lifecycle actions are visually separated.

Exercise Assignments provides compact jump navigation and a distinct container explaining where each selected exercise contributes. Selected exercises render first; alternates remain collapsed until requested. Static tags are neutral; blue is reserved for selected or interactive controls. Candidate `Why the Score?` reports real objective, muscle, equipment, evidence, progression, fatigue, redundancy, and stability inputs. Full Program Review renders separate session cards whose headers show exercise and working-set totals. The normal review shows only unresolved Blocking Issues, Recommended Changes, Warnings, and useful Optional Suggestions. Passed checks, unknown-metadata notices, and non-actionable redundancy observations are hidden. Actionable findings expose `Regenerate with Practical Limits`, which reruns the same 18-set, two-exercise-per-muscle, priority, and recovery constraints.

Program Slot summaries use exercise chips, session badges, role/frequency badges, and omit repeated generic rationale. `Compare Details` compares movement, muscles, equipment, stability, progression, fatigue, placement, and score limitations. Missing metadata is omitted or labelled `Metadata Review Needed`. The weekly summary makes direct sets primary, then taxonomy-defined fractional contribution, weighted stimulus, isometric exposure, target, frequency, and taxonomy version. Direct sets count 1.0; verified fractional work counts 0.5 or 0.25; incidental and unknown relationships count zero; isometric exposure is fatigue-only. `Why the Score?` lists the actual muscle relationships and credits rather than inferring targets from an exercise name.

Readiness capture includes sleep, quality, HRV, resting HR, soreness, illness, affected muscle, nutrition/protein status, and an outside-band note. Guidance uses concise states such as “Go as planned,” monitoring/adjustment, lower-fatigue, or rest guidance. Nutrition is a readiness input, not a standalone food-tracking workflow.

## States and feedback

Privacy/Support return queries and notification/workout deep-link fields are one-shot routing inputs. Once consumed, canonical Today, Plan, Progress, or More navigation removes those fields while retaining unrelated deployment-verification parameters, so reload cannot reopen a stale destination or action.

- **Empty:** Controlled “not enough data,” no-template/history/chart messages explain qualifying data needed; analytics do not substitute unrelated scores.
- **Loading:** Initial persistence/evidence loading occurs before primary use; imports expose attempt-scoped progress/error feedback and remote deletion exposes its continuation state. **NEEDS REVIEW:** there is no unified skeleton/loading design.
- **Error:** Toasts, accessible import validation, retryable/non-retryable remote-deletion status, API status, persistence fallback, and setup errors are surfaced without exposing secrets. When both persistence layers reject an ordinary change, Settings rerenders the recovery guidance immediately while retaining the in-memory setting.
- **Success:** Live-region toast and import status, completed controls, timer-complete notice, notification tests, remote-deletion confirmation, submission summary, PR/grade feedback.
- **Confirmation:** Submit, cancel workout, save/cancel history edits, template deletion, and clear-data flows require explicit action.
- **History-edit persistence, recovery conflicts, and app updates:** Starting an edit first flushes a cloned stable snapshot so an unrelated debounced template or settings change is retained. Startup compares valid IndexedDB and local fallback copies by revision, canonical content, then reliable persistence time. A newer fallback cannot be shadowed by stale readable IndexedDB. Divergent equal-revision copies with equal or unavailable timing are both preserved; Settings shows an always-visible status that explains the conflict, asks for an export, and identifies confirmed **Clear All Local App Data** as the only destructive resolution. Ordinary saves and reloads do not remove that alternate. Confirmed clearing announces completion and returns to fresh defaults only after persistence succeeds; a reload must not restore either the alternate or the conflict. If navigation or another setting changes while the edit snapshot is being saved, edit mode does not open or roll back the newer state; the app awaits a second save of the current clone. The toast says the current changes were saved only after that succeeds. If both stores fail, it instead says the changes are not durable and tells the user to export or retry before reloading. If a new service-worker controller activates while the edit is starting or active, the app stays in place and the update banner asks the user to Save or Discard. Once the edit is resolved, `Update now` is explicit, saves the current stable state, and refuses to reload if both local persistence layers fail.
